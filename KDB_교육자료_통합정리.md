## 서문 / 교육 목표

### 왜 KDB 임직원이 AI를 배워야 하는가?

생성형 AI는 2026년 현재 금융기관의 심사·리서치·보고 업무를 근본적으로 바꾸고 있습니다. KDB와 같은 정책금융기관에게 AI는 **단순 문서 도구가 아니라**, 기업여신 심사·산업분석·PF 타당성·M&A 자문·ESG 평가 등 핵심 업무의 품질과 속도를 동시에 높이는 **업무 파트너**입니다.

### 교육 목표

1. 생성형 AI를 **KDB 실제 업무**(여신심사, 산업분석, PF, M&A, ESG, 시장동향)에 즉시 적용할 수 있는 프롬프트·워크플로 역량 확보
2. 제조/일반 예시가 아닌, **KDB 심사역·애널리스트·딜 매니저의 일상 업무 흐름**을 AI에 탑재
3. 금융권 특유의 **보안·컴플라이언스**(신용정보법, 전자금융감독규정, 사내 LLM 가이드라인) 판단 기준 체득
4. **할루시네이션·규제 오인용**이 실제 여신·투자 의사결정에 미치는 리스크를 직시하고, 검증 3단계 루틴화
5. **에이전트·워크플로우** 개념으로 반복 업무를 자동화하고, Human-in-the-loop 원칙으로 최종 책임을 유지

### 모듈 구성 한눈에

| 모듈 | 제목 | 시간 | 핵심 키워드 |

## Part I — 공통 프레임워크

### 1. RTCF 프레임워크 — 구조적 프롬프트의 뼈대

업무 지시서에 "누가, 무엇을, 어떤 상황에서, 어떤 형태로"가 빠지면 안 되듯, AI 지시서에도 4가지 필수 항목이 있습니다.

| 요소 | 의미 | KDB 업무 비유 | 프롬프트 예시 |
|------|------|--------------|--------------|
| **R**ole (역할) | AI에게 부여하는 전문가 역할 | "보고서 수신자가 기대하는 직무" | "당신은 15년차 기업여신 심사역입니다" |
| **T**ask (과제) | AI에게 요청하는 구체적 작업 | "요청 사항" | "여신 심사의견서 초안을 작성해주세요" |
| **C**ontext (맥락) | 업무 상황, 데이터, 배경 | "배경 및 현황" | "A社 중견기업, 자동차 부품, 시설자금 500억 신청, 담보 공장부지+건물+기계설비" |
| **F**ormat (형식) | 원하는 결과물의 구조와 분량 | "양식 지정" | "재무/업종/담보/리스크 4열 표 + 종합의견 3줄" |

> 🔑 핵심: RTCF 네 개 중 **세 개만 채워도** 결과 품질이 몰라보게 올라갑니다. 매번 완벽한 5요소 프롬프트를 쓸 필요는 없습니다.

### 2. 5대 프롬프트 기법 + 조합 공식

| 기법 | 핵심 원리 | 언제 쓰는가 | 여신심사 예시 | 산업분석/IB 예시 |
|------|----------|------------|----------------|--------------------|
| **Zero-shot** | 예시 없이 바로 요청 | 단순 요약, 번역, 정보 정리 | 일일 여신 동향 요약 | 회의록 요약 |
| **Few-shot** | 예시를 먼저 보여줌 | 양식 통일, 패턴 학습, 반복 업무 | 심사 의견서 양식 맞추기 | 산업 리포트 템플릿 적용 |
| **Role-play** | 전문가 역할 부여 | 전문가 관점이 필요한 분석 | 20년차 심사역으로 리스크 분석 | 수석 애널리스트로 업황 분석 |
| **CoT** | 단계별 추론 요청 | 복잡한 분석, 의사결정 | 대출 심사 리스크 단계별 추론 | PF DSCR 민감도 단계별 추론 |
| **구조화 출력** | 형식 지정(표, 리스트) | 보고서, 비교표, 체크리스트 | 여신심사 의견서 항목별 정리 | Top 3 플레이어 비교표 |

#### 기법 조합 공식

> 🔑 조합 공식 (난이도별)
>
> - **기본 조합** = Role-play + 구조화 출력 → 전문가 관점의 체계적 보고서 (예: 산업 동향 요약표, 여신 점검표)
> - **분석 조합** = Role-play + CoT + 구조화 출력 → 단계별 전문 분석 보고서 (예: 여신등급 하락 원인 단계 분석, PF DSCR 민감도 분석)
> - **최강 조합** = Role-play + Few-shot + CoT + 구조화 출력 → 사내 양식에 맞춘 심층 분석 (예: KDB Monthly 포맷 산업분석, 여신심사위원회 표준 의견서)

#### 상황별 조합 가이드

| 업무 유형 | 권장 기법 조합 | 이유 |
|----------|--------------|------|
| 단순 번역/요약 | Zero-shot + 형식 | 간단하므로 바로 요청, 형식만 지정 |
| 반복 문서 작성 | Few-shot + 구조화 출력 | 기존 양식을 예시로 보여주고 패턴 학습 |
| 전문 분석 보고서 | Role-play + CoT + 구조화 | 전문가 역할 + 단계적 추론 + 깔끔한 형식 |
| 복잡한 의사결정 | Role-play + CoT | 전문가 관점에서 단계별로 논리 전개 |
| 고객사·기업 대응 문서 | Role-play + Few-shot + 형식 | 전문가 톤 + 템플릿 활용 + 일관된 형식 |

### 4. RAG (Retrieval-Augmented Generation) 3단계

AI가 모르는 KDB 내부 데이터·최신 시장 데이터를 연결하는 방법.

| 단계 | 동작 | KDB 적용 |
|------|------|---------|
| **1. 검색 (Retrieval)** | 질문과 관련된 문서를 DB에서 검색 | 내부 심사보고서, 여신 DB, ECOS 미러링, 과거 PF 딜 아카이브에서 검색 |
| **2. 증강 (Augmentation)** | 검색된 문서를 프롬프트에 첨부 | 찾아온 자료를 AI에게 "이 자료를 참고해서 답변해"라고 전달 |
| **3. 생성 (Generation)** | AI가 첨부된 문서 근거로 답변 생성 | 실제 DART·KIS·내부 데이터에 기반한 정확한 분석 |

> 🔑 핵심: RAG는 AI가 "기억에 의존"하는 방식에서 "자료를 찾아보고 답변"하는 방식으로 바꿔주는 기술입니다. **오픈북 시험**처럼, 교재를 보면서 답을 쓰는 것과 같습니다.

### 5. MCP (Model Context Protocol) — "AI 세계의 USB 포트"

| 구분 | 기존 방식 | MCP 방식 |
|------|----------|---------|
| **데이터 전달** | 사용자가 직접 복사-붙여넣기 | AI가 필요한 데이터를 자동으로 가져옴 |
| **시스템 연결** | 시스템마다 별도 개발 필요 | 표준 규격으로 한 번에 연결 |
| **실시간성** | 복사 시점의 데이터만 활용 | 항상 최신 데이터에 접근 |
| **활용 범위** | 텍스트 위주 | DART, FnGuide, 사내 여신DB, Bloomberg, ECOS 등 다양한 시스템 |

#### KDB MCP 활용 시나리오 3가지

1. **사내 여신 DB에서 포트폴리오 분석 자동화**: "지난 분기 2차전지 업종 신규 여신 승인 건을 요약해줘" → MCP가 내부 여신 DB에서 직접 데이터 추출 → 포트폴리오 분석 + 이상 패턴 감지
2. **DART + 내부 심사보고서 동시 참조**: "A社 재무와 당행 과거 심사 의견을 함께 보여줘" → MCP가 DART 공시 + 내부 심사 아카이브 동시 조회
3. **Bloomberg·ECOS 연동 실시간 브리핑**: "오늘 기준금리·회사채 스프레드·원/달러 환율" → MCP가 Bloomberg·ECOS 실시간 스냅 → 자동 요약

> ⚠ 보안 원칙: MCP는 강력한 연결인 만큼 **최소 권한 원칙(Least Privilege)**이 필수. AI가 접근할 수 있는 데이터의 범위를 업무별로 명확히 설정. 고객 신용정보·미공개 딜 정보는 사내 LLM 한정.

### 6. 에이전틱 워크플로우 — AI 팀원 설계

#### 단순 프롬프트 vs 에이전틱 워크플로우

| 구분 | 단순 프롬프트 | 에이전틱 워크플로우 |
|------|-------------|------------------|
| 동작 방식 | 질문 1개 → 답변 1개 | AI가 계획 → 실행 → 검증을 자동 반복 |
| 사람의 개입 | 매 단계마다 사용자가 다음 질문 입력 | AI가 자율적으로 다음 단계 진행, 핵심 판단만 사람 |
| 적합한 업무 | 간단한 질의응답, 단문 작성 | 복잡한 분석, 다단계 보고서, 종합 리서치 |
| KDB 예시 | "2차전지 업종 리스크 3개 알려줘" | 여신 후보 추출 → 재무분석 → 업종비교 → 심사메모 → 자체검토 |

#### 에이전트 설계 3단계

1. **업무 목표 정의**: 최종 결과물(심사의견서, PF 타당성 보고서 등)을 명확히
2. **단계 분해**: 목표 달성에 필요한 작업을 순서대로 나열
3. **역할 부여**: 각 단계에 가장 적합한 전문가 역할 부여

#### 예시 1 — 여신심사 에이전트 팀 (KDB)

| 에이전트 | 역할 | 수행 업무 |
|---------|------|---------|
| 재무분석 에이전트 | 재무분석 전문가 | DART 재무제표 추출·비율 계산 |
| 업종분석 에이전트 | 산업 애널리스트 | 해당 업종 동향·경쟁구도 분석 |
| 담보평가 에이전트 | 담보평가 전문가 | 감정평가서 검토·LTV 산정 |
| 통합 에이전트 | 심사역 | 위 3개 결과를 심사의견서로 통합 |

#### 예시 2 — PF 타당성 에이전트 팀 (KDB)

| 에이전트 | 역할 | 수행 업무 |
|---------|------|---------|
| 시장분석 에이전트 | 산업·시장 전문가 | 수요 전망, 정책 환경, 경쟁 딜 |
| CF모델링 에이전트 | 재무 모델러 | 연도별 CF, DSCR/LLCR 산정 |
| 리스크평가 에이전트 | 리스크 전문가 | 건설·운영·정책·환 리스크 매트릭스 |
| 종합 에이전트 | PF 심사역 | 전체 종합 및 위원회 상정 초안 |

#### 워크플로우 설계 3원칙

- **분해**: 큰 업무를 작고 명확한 단계로 (각 단계 입·출력 명확)
- **자동화**: 앞 단계 결과가 다음 단계 입력이 되도록 연결
- **검증**: 중간·최종 검증 포함

#### Human-in-the-loop — 사람이 반드시 참여해야 하는 이유

1. AI는 여전히 할루시네이션을 일으킬 수 있음
2. 업무 맥락·회사 상황의 종합 판단은 사람만이 가능
3. 최종 결과물 책임은 항상 사람에게 있음

> 🔑 핵심: "일상 작업은 AI가, 사람은 판단·의사결정에 집중." 에이전트 설계의 첫걸음은 **"이 업무를 어떤 전문가 팀에게 맡길 것인가?"**를 생각하는 것입니다.

### 7. 할루시네이션 방지 3단계 팩트체크

금융권 업무에서 AI의 허위 정보는 즉시 법적·재무적 리스크가 됩니다.

| 단계 | 무엇을 | KDB 검증 방법 |
|------|-------|-------------|
| **Step 1: 출처 존재 확인** | AI가 인용한 법령·논문·판례·공시가 실존하는가 | 국가법령정보센터, DART, 대법원 종합법률정보, RISS, ECOS에서 직접 검색 |
| **Step 2: 원문 대조** | 조항 번호·수치·기준일이 원문과 일치하는가 | 원문 PDF와 AI 요약을 나란히 비교, Claude에게 "두 텍스트 차이점 찾기" 프롬프트 활용 |
| **Step 3: 전문가 검증** | 법무팀·심사부·리스크관리부의 최종 확인 | 결재 상정 전 담당 부서 리뷰, 면책 문구 필수 삽입 |

> ⚠ 주의: "AI가 확인해줬으니 맞겠지"는 가장 위험한 판단. AI의 자신감 있는 어투에 속지 마세요. **최종 책임은 항상 사람에게 있습니다.**

|------|-----|------|

```
아래 재무·담보 데이터를 바탕으로 팀장 보고용 심사 의견서를 작성해주세요.

[심사 데이터]
- 매출채권회전율: 5.2회 (업종평균 7.0회) → 미흡
- DSCR (최근 4개 분기 평균): 1.12배 (기준 1.3배 이상) → 미흡
- 부채비율: 215% (직전 198% 대비 상승) → 주의
- 담보: 공장부지·건물 감정가 310억, 1순위 설정 가능
- 신용등급(NICE): A-(안정적), 12개월 이내 변동 없음

형식: 아래 표 구조로 작성
| 심사항목 | 측정값 | 기준값 | 판정 | 조치/의견 | 점검주기 |

특이사항 및 종합 의견을 3줄 이내로 추가해주세요.
```

#### 5단계 프롬프트 체크리스트

| 단계 | 체크 항목 | 여신심사 예시 | 산업분석/PF 예시 |

| 4️⃣ 형식 | 표, 리스트, 보고서 | "5개 항목별 구조화" | "비교표 + 신호등 + 의견" |

| **Copilot** | MS365(엑셀·PPT·Outlook) 통합 | 창의성 낮음 | 사내 엑셀 자동화, PPT 초안 | 라이센스 조직에서만 |
| **사내 LLM (KDB)** | 보안·내부자료 연결 | 외부 모델 대비 범용성 부족 | 여신 포트폴리오 분석, 내부 보고서 RAG | 외부 LLM 대비 성능 차이 — 용도 분리 |

#### 보안 판단 플로차트 — 외부 AI vs 사내 LLM

```
데이터를 외부 AI에 입력하려 하는가?
├── 공개 데이터인가? (뉴스, 공시, 공개 통계)
│   └── YES → 외부 AI 사용 가능
└── 비공개 데이터인가? (고객 정보, 내부 문서, 재무 상세)
    ├── 식별 정보를 제거(마스킹)할 수 있는가?
    │   ├── YES → 마스킹 후 외부 AI (조심스럽게)
    │   └── NO → 사내 LLM만 사용
    └── 심사 중 딜, 미공개 M&A, PF 입찰 정보인가?
        └── 무조건 사내 LLM만, 외부 입력 절대 금지
```

#### 금융권 사내 LLM이 필수인 이유

1. **신용정보법**: 개인신용정보·기업신용정보의 외부 처리 제한

4. **내부 데이터 활용**: 사내 LLM은 내부 DB·문서에 RAG 연결 가능

### 모듈 3 — 보고서 작성 실전 가이드

> 45분 · AI 보고서 5단계 워크플로, 여신·시장·산업 템플릿, Few-shot, 자동화

#### 학습 목표

1. AI 보고서 작성의 5단계 워크플로(목적 정의 → 자료 수집 → 초안 작성 → 검증 → 다듬기)를 이해하고 실행할 수 있다
2. KDB 업무별(여신·시장·산업·PF·리스크·포트폴리오) 보고서 템플릿 6종을 익혀 즉시 활용할 수 있다
3. Few-shot 기법으로 사내 양식에 맞춘 일관된 보고서를 생성할 수 있다
4. 반복 보고서(일일·주간·월간)의 자동화 워크플로를 설계할 수 있다

#### 5단계 워크플로 개요

| 단계 | 주체 | 주요 활동 |
|------|------|----------|
| 1️⃣ 목적 정의 | 사람 | 보고서 유형, 독자, 목적, 핵심 메시지 결정 |
| 2️⃣ 자료 수집 | AI + 사람 | DART, FnGuide, Bloomberg, 내부 DB에서 데이터 수집 |
| 3️⃣ 초안 작성 | AI 주도 | 프롬프트 + Few-shot + 구조화 출력으로 초안 생성 |
| 4️⃣ 검증 | 사람 + AI | 수치·법규·출처 3단계 팩트체크 |
| 5️⃣ 다듬기 | 사람 | 톤 조정, 경영진 관점 반영, 면책 문구 삽입 |

### 모듈 5 — 할루시네이션 방지와 팩트체크

> 35분 · 할루시네이션 유형, 팩트체크 3단계, 10대 체크리스트, RAG/Grounding, 금융권 보안

#### 학습 목표

1. 할루시네이션의 정의, 유형, 발생 원인을 이해하고 AI 답변의 경고 신호를 감지할 수 있다
2. AI가 생성한 수치·법규·판례·출처를 3단계로 팩트체크할 수 있다
3. 보고서 작성 시 적용할 수 있는 10개 체크리스트를 활용해 할루시네이션을 제거할 수 있다
4. RAG, Grounding, 출처 표시 등 기술적 대응 방법을 이해하고 선택적으로 적용할 수 있다

#### 할루시네이션의 4가지 유형

| 유형 | 설명 | KDB 예시 |
|------|-----|---------|
| **허위 사실 (Fabrication)** | 없는 정보를 만들어냄 | "자본시장법 제300조 ○항"이 실제 존재하지 않음 |
| **왜곡 (Distortion)** | 실제 정보를 잘못 표현 | A社 신용등급을 "AA"로 보고(실제 A) |
| **오인용 (Misattribution)** | 출처·작성자를 틀리게 인용 | 실제로 없는 판결문 사건번호 생성 |
| **시점 오류 (Temporal Error)** | 시점을 잘못 기재 | "바젤 IV 2026년 전면 시행"(실제 일정 상이) |

#### 할루시네이션 경고 신호

- AI가 지나치게 구체적인 수치(예: "2026년 3월 15일에 제정된...")를 자신 있게 제시
- 출처를 요구하면 애매한 답변 또는 없는 URL
- 같은 질문을 다시 물으면 답변이 미세하게 바뀜
- 상식적으로 가능성이 낮은 조합(예: 특정 기업의 허위 재무 수치)

#### 팩트체크 3단계 (Step 1)

```
[Step 1 — 출처 존재 확인]
1. AI 인용 법령 → law.go.kr, 금감원 법규
2. AI 인용 논문 → Google Scholar, RISS, SSRN
3. AI 인용 판례 → 대법원 종합법률정보
4. AI 인용 기업 재무 → DART 공시 원문
5. AI 인용 신용등급 → KIS Value, NICE 원본 리포트
6. AI 인용 금리·환율 → 한은 ECOS
7. AI 인용 산업 통계 → KIET, 통계청, KDB 미래전략연구소
8. AI 인용 뉴스 → 해당 언론사 사이트에서 직접 검색
```

#### 팩트체크 3단계 (Step 2) — 원문 대조

```
[대조 체크리스트]
□ 조항 번호(법령) / 공시 번호(DART) / 리포트 ID(KIS·NICE) 정확성
□ 조항·공시·리포트의 내용이 AI 요약과 일치
□ 수치(재무, 신용지표, 금리, 환율, 금액)가 원문과 같음
□ 적용 대상(업종, 기업, 기간) 올바름
□ 시행일·공시일·기준일 최신
```

#### 팩트체크 3단계 (Step 3) — 전문가 검증

| 검증 대상 | 검증 주체 |
|----------|---------|
| 법률·규제 조항 | 준법감시실 / 법무팀 |
| 기업 재무 데이터 | 여신심사부 / 리스크관리부 |
| 신용등급 · 신용평가 의견 | 여신심사부 / 리스크관리부 |
| 금리 · 환율 · 경제지표 | 리서치부 / 자금부 |
| PF 사업성 · DSCR 가정 | 프로젝트금융부 |
| ESG 평가 · 탄소 데이터 | 지속가능경영실 |

#### 10대 할루시네이션 제거 체크리스트 (결재 상정 전)

| # | 점검 항목 |
|---|----------|

#### 기술적 대응 — RAG / Grounding / 출처 표시

- **RAG**: 내부 심사보고서·여신 DB·ECOS 미러링·내규를 AI에 연결 → 허위 생성 원천 차단
- **Grounding**: AI 답변을 특정 문서에 근거하도록 제한 (Claude의 Projects 기능, Gemini Grounding)
- **Perplexity 출처 인용**: 모든 주장에 인라인 링크 → 클릭으로 원문 즉시 확인

**Before — 나쁜 프롬프트:**
```
A사 대출 심사해줘.
```
→ 교과서적 일반론

**After — 좋은 프롬프트 (전체):**
```
당신은 KDB 여신심사부 20년차 선임 심사역입니다.
자동차 부품 업종 여신 심사 경험 150건 이상, 산업분석부 파견
경력 3년 보유. 중견기업 시설자금 심사가 전문 영역입니다.

아래 실제 데이터를 기반으로 심사보고서 초안을 작성해주세요.

[사내 시스템에서 추출한 실제 데이터 — 붙여넣기]
- A사 최근 3개년 재무제표 (손익/재무상태/현금흐름)
- 담보 감정평가 요약 (사업용 부지 XX억, 건물 XX억, 기계설비 XX억)
- 업종: 자동차 부품 (Tier 1/2 여부, 주요 매출처)

분석 순서는 아래 CoT(단계별 사고)를 따라주세요:

Step 1 — 재무 분석
  - 수익성(영업이익률, ROE, ROA)
  - 안정성(부채비율, 이자보상배율, 유동비율)
  - 성장성(매출/영업이익 CAGR)
  - 현금흐름(영업CF, FCF, CFADS)
  - 3개년 추이와 경쟁사 벤치마크 필요 항목 표시

Step 2 — 업종 분석
  - 자동차 부품 업종 사이클 국면
  - 전기차 전환 영향 (내연기관 의존도)
  - 주요 매출처 집중도 리스크

Step 3 — 담보 분석
  - 담보별 감정가 및 유효담보가치
  - LTV 산정 (대출금 500억 기준)
  - 담보 유동성 평가

Step 4 — 종합 신용등급 매트릭스
  - 정량(재무) + 정성(업종/경영/담보) 가중치 적용
  - KDB 내부 신용등급 구간 제안 (BB+/BBB-/BBB 등 후보)
  - 최종 등급은 "심사위원회 결정사항"으로 표기

출력 형식:
| 섹션 | 핵심 분석 | 리스크 요인 | 추가 확인 필요 |

출력 조건:
- 과거 심사한 유사 딜 양식(Few-shot으로 제공)을 따라주세요
- 신용등급을 확정하지 말고 "등급 후보"로만 제시
- 여신 조건(금리/만기/약정) 제안 포함
- 모든 수치는 입력된 실제 데이터 기반으로만 계산
- 업종 데이터가 없는 경우 "KIS Value/FnGuide 확인 필요"로 표시
```

**기법 해설:** Role-play + CoT + 구조화 출력 + Few-shot 최강 조합.

**검증 포인트:**
- DART 반기/사업보고서와 재무 수치 대조
- KIS·NICE 신용등급 추이
- 담보 감정평가서 원문

**보안 판단:**
- 기업명·대표자명 등 민감정보는 **사내 LLM** 또는 **마스킹 후 외부 LLM**

### C. 금융권 보안 의사결정 트리

- YES → 외부 LLM 사용 가능 (ChatGPT, Claude, Gemini)
- NO → Q2

**Q2. 식별 정보(기업명, 대표자명, 계좌번호 등)를 마스킹할 수 있는가?**
- YES → 마스킹 후 외부 LLM 조심스럽게 사용
- NO → Q3

**Q3. 미공개 딜 정보(M&A, PF 입찰, 심사 중 여신)인가?**
- YES → **무조건 사내 LLM만** 사용, 외부 입력 절대 금지
- NO → Q4

**Q4. 고객 신용정보·개인정보 포함인가?**
- YES → 사내 LLM + RAG + 권한 관리 필수
- NO → 재검토 후 마스킹 여부 결정

### E. 실습 문제 세트

#### 모듈 1 실습

1. "여신 심사 보고서 써줘" 프롬프트의 3가지 함정을 식별하고 개선 버전 작성
2. A사 중견 화학기업 DSCR 하락 원인 분석용 CoT 프롬프트 작성 (5단계)
3. 본인 업무 1개 선정 → RTCF 프레임워크로 템플릿화

#### 모듈 2 실습

1. 동일 질문("2차전지 산업 2026 전망")을 ChatGPT / Claude / Perplexity에 각각 입력 → 결과 차이 비교표 작성
2. 보안 판단 플로차트로 본인 업무 데이터 3종 분류
3. 사내 LLM으로 여신 포트폴리오 분석 프롬프트 초안 작성

#### 모듈 3 실습

1. 본인 부서 표준 보고서 양식을 Few-shot 예시로 → AI에 적용 테스트
2. 일일 시장 브리핑 자동화 워크플로 설계
3. PF 타당성 검토 템플릿을 실제 딜 건에 적용해 초안 생성

#### 모듈 4 실습

1. 가상 상황("A사 자본시장법 위반 가능성 여부")을 Perplexity로 검색 → 출처 3개 원문 확인
2. K-Taxonomy + TCFD 교차로 2차전지 기업 ESG 자격 검증
3. 해외 PF 딜 규제 체크리스트 AI 초안 → 원문으로 3개 이상 검증

#### 모듈 5 실습

1. AI가 제시한 가상 법령 3건 중 실존 1건, 허위 2건 구별
2. DART 원문 1건 + AI 요약 1건 → 차이점 매트릭스 작성
3. 본인 최근 보고서 1건에 10대 체크리스트 전수 적용

#### 모듈 6 실습

1. 시나리오 1 프롬프트를 본인 업무 차주로 치환 → 실제 생성 결과 비교
2. 시나리오 2를 2026년 반도체 산업으로 치환 → KDB Monthly 초안
3. 시나리오 3을 태양광 PF 300MW로 치환 → DSCR 민감도 매트릭스

### F. 용어 사전

#### AI / 프롬프트 기법

| 용어 | 의미 |
|------|-----|
| **RTCF** | Role-Task-Context-Format. 구조적 프롬프트 4요소 |
| **Zero-shot** | 예시 없이 바로 요청하는 프롬프트 기법 |
| **Few-shot** | 예시 1~3개를 먼저 보여주는 프롬프트 기법 |
| **Role-play** | AI에게 전문가 역할을 부여하는 기법 |
| **CoT** | Chain of Thought. 단계별 추론을 요청하는 기법 |
| **RAG** | Retrieval-Augmented Generation. 외부 문서 검색 후 증강 생성 |
| **MCP** | Model Context Protocol. AI와 외부 시스템의 표준 연결 규격 |
| **Grounding** | AI 답변을 특정 자료에 근거하도록 제한하는 기술 |
| **Function Calling** | AI가 사전 정의된 함수를 호출하여 실행하는 기능 |
| **Agentic Workflow** | AI가 계획·실행·검증을 자율적으로 반복하는 워크플로 |
| **Human-in-the-loop** | 최종 판단·승인에 사람이 참여하는 구조 |

#### KDB 업무

| 용어 | 의미 |
|------|-----|
| **KDB Monthly** | KDB 발간 월간 산업·경제 분석 리포트 |
| **DSCR** | Debt Service Coverage Ratio. 원리금 상환 비율 |
| **LLCR** | Loan Life Coverage Ratio. 대출 기간 커버리지 비율 |
| **CFADS** | Cash Flow Available for Debt Service |
| **LTV** | Loan to Value. 담보가치 대비 대출 비율 |
| **NPL** | Non-Performing Loan. 부실채권 |
| **PF** | Project Finance. 프로젝트 파이낸스 |
| **PPA** | Power Purchase Agreement. 전력구매계약 |
| **REC** | Renewable Energy Certificate. 신재생에너지 공급인증서 |
| **SLL** | Sustainability-Linked Loan. 지속가능연계대출 |
| **Covenant** | 여신 약정에 포함되는 재무 조건 |
| **DART** | 금융감독원 전자공시 시스템 |
| **ECOS** | 한국은행 경제통계시스템 |
| **KIS / NICE** | 한국신용평가 / NICE평가정보. 국내 2대 신용평가사 |
| **K-Taxonomy** | 한국형 녹색분류체계 |
| **TCFD** | Task Force on Climate-related Financial Disclosures |
| **IRA** | Inflation Reduction Act. 미국 인플레이션 감축법 |
| **CBAM** | Carbon Border Adjustment Mechanism. EU 탄소국경조정제도 |
