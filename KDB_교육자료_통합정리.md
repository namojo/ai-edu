## 서문 / 교육 목표

### 왜 KDB생명 임직원이 AI를 배워야 하는가?

생성형 AI는 2026년 현재 보험업의 인수심사·보험금심사·자산운용·상품개발 업무를 근본적으로 바꾸고 있습니다. KDB생명과 같은 생명보험사에게 AI는 **단순 문서 도구가 아니라**, 보험 인수심사·보험시장 분석·자산운용 투자검토·상품개발·리스크관리 등 핵심 업무의 품질과 속도를 동시에 높이는 **업무 파트너**입니다.

### 교육 목표

1. 생성형 AI를 **KDB생명 실제 업무**(인수심사, 보험금심사, 자산운용, 상품개발, 리스크관리, 고객서비스)에 즉시 적용할 수 있는 프롬프트·워크플로 역량 확보
2. 제조/일반 예시가 아닌, **KDB생명 언더라이터·보험계리사·자산운용 담당자의 일상 업무 흐름**을 AI에 탑재
3. 보험업 특유의 **보안·컴플라이언스**(보험업법, 개인정보보호법, K-ICS, IFRS 17, 사내 LLM 가이드라인) 판단 기준 체득
4. **할루시네이션·규제 오인용**이 실제 인수심사·투자 의사결정에 미치는 리스크를 직시하고, 검증 3단계 루틴화
5. **에이전트·워크플로우** 개념으로 반복 업무를 자동화하고, Human-in-the-loop 원칙으로 최종 책임을 유지

## Part I — 공통 프레임워크

### 1. RTCF 프레임워크 — 구조적 프롬프트의 뼈대

업무 지시서에 "누가, 무엇을, 어떤 상황에서, 어떤 형태로"가 빠지면 안 되듯, AI 지시서에도 4가지 필수 항목이 있습니다.

| 요소 | 의미 | KDB생명 업무 비유 | 프롬프트 예시 |
|------|------|--------------|--------------|
| **R**ole (역할) | AI에게 부여하는 전문가 역할 | "보고서 수신자가 기대하는 직무" | "당신은 15년차 보험 언더라이터입니다" |
| **T**ask (과제) | AI에게 요청하는 구체적 작업 | "요청 사항" | "고액 건강보험 인수심사 의견서 초안을 작성해주세요" |
| **C**ontext (맥락) | 업무 상황, 데이터, 배경 | "배경 및 현황" | "피보험자 50세 남성, 기존 고혈압 병력, 보험금액 5억원, 직업: 사무직" |
| **F**ormat (형식) | 원하는 결과물의 구조와 분량 | "양식 지정" | "건강상태/직업위험/재정상태/종합판정 4열 표 + 인수조건 의견 3줄" |

> 🔑 핵심: RTCF 네 개 중 **세 개만 채워도** 결과 품질이 몰라보게 올라갑니다. 매번 완벽한 5요소 프롬프트를 쓸 필요는 없습니다.

### 2. 5대 프롬프트 기법 + 조합 공식

| 기법 | 핵심 원리 | 언제 쓰는가 | 인수심사 예시 | 보험시장분석/자산운용 예시 |
|------|----------|------------|----------------|--------------------|
| **Zero-shot** | 예시 없이 바로 요청 | 단순 요약, 번역, 정보 정리 | 일일 보험시장 동향 요약 | 회의록 요약 |
| **Few-shot** | 예시를 먼저 보여줌 | 양식 통일, 패턴 학습, 반복 업무 | 인수심사 의견서 양식 맞추기 | 보험시장 분석 리포트 템플릿 적용 |
| **Role-play** | 전문가 역할 부여 | 전문가 관점이 필요한 분석 | 20년차 언더라이터로 리스크 분석 | 수석 보험계리사로 상품 분석 |
| **CoT** | 단계별 추론 요청 | 복잡한 분석, 의사결정 | 인수심사 리스크 단계별 추론 | 자산운용 투자건 수익률 민감도 단계별 추론 |
| **구조화 출력** | 형식 지정(표, 리스트) | 보고서, 비교표, 체크리스트 | 인수심사 의견서 항목별 정리 | 경쟁 보험상품 비교표 |

#### 기법 조합 공식

> 🔑 조합 공식 (난이도별)
>
> - **기본 조합** = Role-play + 구조화 출력 → 전문가 관점의 체계적 보고서 (예: 보험시장 동향 요약표, 인수심사 점검표)
> - **분석 조합** = Role-play + CoT + 구조화 출력 → 단계별 전문 분석 보고서 (예: 손해율 악화 원인 단계 분석, 자산운용 수익률 민감도 분석)
> - **최강 조합** = Role-play + Few-shot + CoT + 구조화 출력 → 사내 양식에 맞춘 심층 분석 (예: KDB생명 보험시장 분석 리포트, 인수심사위원회 표준 의견서)

#### 상황별 조합 가이드

| 업무 유형 | 권장 기법 조합 | 이유 |
|----------|--------------|------|
| 단순 번역/요약 | Zero-shot + 형식 | 간단하므로 바로 요청, 형식만 지정 |
| 반복 문서 작성 | Few-shot + 구조화 출력 | 기존 양식을 예시로 보여주고 패턴 학습 |
| 전문 분석 보고서 | Role-play + CoT + 구조화 | 전문가 역할 + 단계적 추론 + 깔끔한 형식 |
| 복잡한 의사결정 | Role-play + CoT | 전문가 관점에서 단계별로 논리 전개 |
| 고객·계약자 대응 문서 | Role-play + Few-shot + 형식 | 전문가 톤 + 템플릿 활용 + 일관된 형식 |

### 4. RAG (Retrieval-Augmented Generation) 3단계

AI가 모르는 KDB생명 내부 데이터·최신 보험시장 데이터를 연결하는 방법.

| 단계 | 동작 | KDB생명 적용 |
|------|------|---------|
| **1. 검색 (Retrieval)** | 질문과 관련된 문서를 DB에서 검색 | 보험계약DB, 보험금지급DB, 언더라이팅 기록, 과거 투자건 아카이브에서 검색 |
| **2. 증강 (Augmentation)** | 검색된 문서를 프롬프트에 첨부 | 찾아온 자료를 AI에게 "이 자료를 참고해서 답변해"라고 전달 |
| **3. 생성 (Generation)** | AI가 첨부된 문서 근거로 답변 생성 | 실제 DART·보험개발원·내부 데이터에 기반한 정확한 분석 |

> 🔑 핵심: RAG는 AI가 "기억에 의존"하는 방식에서 "자료를 찾아보고 답변"하는 방식으로 바꿔주는 기술입니다. **오픈북 시험**처럼, 교재를 보면서 답을 쓰는 것과 같습니다.

### 5. MCP (Model Context Protocol) — "AI 세계의 USB 포트"

| 구분 | 기존 방식 | MCP 방식 |
|------|----------|---------|
| **데이터 전달** | 사용자가 직접 복사-붙여넣기 | AI가 필요한 데이터를 자동으로 가져옴 |
| **시스템 연결** | 시스템마다 별도 개발 필요 | 표준 규격으로 한 번에 연결 |
| **실시간성** | 복사 시점의 데이터만 활용 | 항상 최신 데이터에 접근 |
| **활용 범위** | 텍스트 위주 | DART, 보험개발원, 보험계약DB, 보험금지급DB, Bloomberg, 금감원 보험통계 등 다양한 시스템 |

#### KDB생명 MCP 활용 시나리오 3가지

1. **보험계약DB에서 포트폴리오 분석 자동화**: "지난 분기 건강보험 신계약 현황과 해약률 추이를 요약해줘" → MCP가 보험계약DB에서 직접 데이터 추출 → 포트폴리오 분석 + 이상 패턴 감지
2. **DART + 언더라이팅 기록 동시 참조**: "A 피보험자의 건강검진 결과와 과거 인수심사 이력을 함께 보여줘" → MCP가 건강검진 데이터 + 인수심사 아카이브 동시 조회
3. **Bloomberg·금감원 보험통계 연동 실시간 브리핑**: "오늘 기준금리·회사채 스프레드·보험업 K-ICS 평균비율" → MCP가 Bloomberg·금감원 보험통계 실시간 스냅 → 자동 요약

> ⚠ 보안 원칙: MCP는 강력한 연결인 만큼 **최소 권한 원칙(Least Privilege)**이 필수. AI가 접근할 수 있는 데이터의 범위를 업무별로 명확히 설정. 계약자 의료정보·미출시 상품 정보는 사내 LLM 한정.

### 6. 에이전틱 워크플로우 — AI 팀원 설계

#### 단순 프롬프트 vs 에이전틱 워크플로우

| 구분 | 단순 프롬프트 | 에이전틱 워크플로우 |
|------|-------------|------------------|
| 동작 방식 | 질문 1개 → 답변 1개 | AI가 계획 → 실행 → 검증을 자동 반복 |
| 사람의 개입 | 매 단계마다 사용자가 다음 질문 입력 | AI가 자율적으로 다음 단계 진행, 핵심 판단만 사람 |
| 적합한 업무 | 간단한 질의응답, 단문 작성 | 복잡한 분석, 다단계 보고서, 종합 리서치 |
| KDB생명 예시 | "건강보험 손해율 추이 알려줘" | 인수심사 대상 추출 → 건강·직업 리스크 분석 → 재정 적정성 평가 → 인수의견서 → 자체검토 |

#### 에이전트 설계 3단계

1. **업무 목표 정의**: 최종 결과물(인수심사 의견서, 투자검토 보고서 등)을 명확히
2. **단계 분해**: 목표 달성에 필요한 작업을 순서대로 나열
3. **역할 부여**: 각 단계에 가장 적합한 전문가 역할 부여

#### 예시 1 — 보험 인수심사 에이전트 팀 (KDB생명)

| 에이전트 | 역할 | 수행 업무 |
|---------|------|---------|
| 의적심사 에이전트 | 의료 리스크 전문가 | 건강검진 데이터 분석·기왕증 평가·질병 리스크 산정 |
| 직업심사 에이전트 | 직업 리스크 전문가 | 직업 위험등급 분류·환경 리스크 평가 |
| 재정심사 에이전트 | 재정 적정성 전문가 | 소득 대비 보험금액 적정성·도덕적 위험 점검 |
| 통합 에이전트 | 수석 언더라이터 | 위 3개 결과를 인수심사 의견서로 통합 |

#### 예시 2 — 자산운용 투자검토 에이전트 팀 (KDB생명)

| 에이전트 | 역할 | 수행 업무 |
|---------|------|---------|
| 시장분석 에이전트 | 투자시장 전문가 | 시장 동향, 금리 환경, 경쟁 투자건 |
| 수익분석 에이전트 | 재무 모델러 | 연도별 현금흐름, 수익률 산정, 민감도 분석 |
| ALM 에이전트 | 리스크 전문가 | 자산-부채 매칭, K-ICS 영향, 금리 리스크 |
| 종합 에이전트 | 투자심사 담당 | 전체 종합 및 투자심의위원회 상정 초안 |

#### 워크플로우 설계 3원칙

- **분해**: 큰 업무를 작고 명확한 단계로 (각 단계 입·출력 명확)
- **자동화**: 앞 단계 결과가 다음 단계 입력이 되도록 연결
- **검증**: 중간·최종 검증 포함

#### Human-in-the-loop — 사람이 반드시 참여해야 하는 이유

1. AI는 여전히 할루시네이션을 일으킬 수 있음
2. 업무 맥락·회사 상황의 종합 판단은 사람만이 가능
3. 최종 결과물 책임은 항상 사람에게 있음

> 🔑 핵심: "일상 작업은 AI가, 사람은 판단·의사결정에 집중." 에이전트 설계의 첫걸음은 **"이 업무를 어떤 전문가 팀에게 맡길 것인가?"**를 생각하는 것입니다.

### 7. 할루시네이션 방지 3단계 팩트체크

보험업 업무에서 AI의 허위 정보는 즉시 법적·재무적 리스크가 됩니다.

| 단계 | 무엇을 | KDB생명 검증 방법 |
|------|-------|-------------|
| **Step 1: 출처 존재 확인** | AI가 인용한 법령·논문·판례·약관·통계가 실존하는가 | 국가법령정보센터, DART, 보험개발원, 생명보험협회, 금감원 보험통계에서 직접 검색 |
| **Step 2: 원문 대조** | 조항 번호·수치·기준일이 원문과 일치하는가 | 원문 PDF와 AI 요약을 나란히 비교, Claude에게 "두 텍스트 차이점 찾기" 프롬프트 활용 |
| **Step 3: 전문가 검증** | 인수심사부·보험금심사부·리스크관리부의 최종 확인 | 결재 상정 전 담당 부서 리뷰, 면책 문구 필수 삽입 |

> ⚠ 주의: "AI가 확인해줬으니 맞겠지"는 가장 위험한 판단. AI의 자신감 있는 어투에 속지 마세요. **최종 책임은 항상 사람에게 있습니다.**

- ✅ 업종 일반론, 규제 해설

#### 관련 법령

- 금융소비자보호법, 전자금융감독규정
- 금감원 AI 활용 가이드라인
- KDB생명 내부통제 기준, 정보보호 지침

### 모듈 1 — 프롬프트 엔지니어링 기초와 실전

> 40분 · 프롬프트 설계, Zero/Few-shot, Role-play, CoT, 구조화 출력, RTCF, 기법 조합

#### 학습 목표

1. 프롬프트 실패의 3가지 원인(모호한 지시, 맥락 부재, 형식 미지정)을 진단할 수 있다
2. 5가지 핵심 프롬프트 기법(Zero-shot, Few-shot, Role-play, CoT, 구조화 출력)을 상황에 맞게 선택할 수 있다
3. 5단계 프롬프트 체크리스트를 활용하여 즉시 업무에 적용할 수 있다
4. 프롬프트 품질에 따른 결과 차이를 체감하고, 기법 조합 공식을 스스로 설계할 수 있다

#### 프롬프트 실패의 3가지 원인

| 함정 | 현상 | 해결 |
|------|-----|------|
| **모호한 지시** | "인수심사 보고서 써줘" → 교과서 같은 일반론 | 구체적 행동·대상·기준을 지시 |
| **맥락 부재** | 피보험자 정보·보험상품·보험금액·건강상태 없이 요청 | 배경·데이터·비교 대상 모두 주입 |
| **형식 미지정** | 긴 에세이로 나와 다시 정리해야 함 | "표", "bullet", "1페이지 이내" 지정 |

#### Before/After 예시 1 — 보험 인수심사 의견서

**Before (나쁜 예):**
```
보험 인수심사 의견서 작성해줘.
```
→ 일반론 나열, 구체적 피보험자 정보 없음, 인수심사 의견서로 활용 불가

**After (좋은 예):**
```
당신은 20년차 보험 언더라이터입니다.

50세 남성 피보험자의 2026년 4월 건강보험 보험금액 5억원 신규 계약에
대해 인수심사합니다. 접수일: 4월 7일, 담당 언더라이터: 김OO 차장.

아래 건강·직업·재정 데이터를 바탕으로 인수심사위원회 보고용 의견서를 작성해주세요.

[심사 데이터]
- 건강상태: 고혈압(약물 조절 중, 3년), BMI 27.5, 최근 건강검진 정상
- 직업: 사무직(위험등급 1급), 위험 취미 없음
- 기존 보유계약: 종신보험 3억원 (타사)
- 연소득: 1.2억원, 순자산: 8억원
- 가족력: 부친 당뇨, 모친 건강

형식: 아래 표 구조로 작성
| 심사항목 | 평가내용 | 기준 | 판정 | 조건·의견 | 비고 |

특이사항 및 종합 인수의견을 3줄 이내로 추가해주세요.
```

#### 5단계 프롬프트 체크리스트

| 단계 | 체크 항목 | 인수심사 예시 | 보험시장분석/자산운용 예시 |
|------|----------|----------------|--------------------|
| 1️⃣ 역할 | "당신은 ___입니다" | "20년차 보험 언더라이터" | "수석 보험계리사" |
| 2️⃣ 맥락 | 배경, 목적, 대상 | "50세 남성, 건강보험 5억원 인수심사" | "건강보험 시장 2026년 전망" |
| 3️⃣ 지시 | 원하는 작업 명확히 | "인수심사 의견서 작성" | "경쟁 3사 보험상품 비교 분석 및 의견" |
| 4️⃣ 형식 | 표, 리스트, 보고서 | "5개 항목별 구조화" | "비교표 + 신호등 + 의견" |
| 5️⃣ 조건 | 분량, 톤, 포함/제외 | "A4 2페이지, 심사위원회 수준" | "언더라이터가 이해하기 쉬운 용어" |

#### 범용 프롬프트 템플릿 (즐겨찾기 저장)

```
당신은 [역할/전문분야] 전문가입니다.
[배경 상황을 2~3문장으로 설명]

다음을 수행해주세요: [구체적 작업 지시]

```

└── 비공개 데이터인가? (계약자 정보, 내부 문서, 심사 상세)
    ├── 식별 정보를 제거(마스킹)할 수 있는가?

#### 5단계 워크플로 개요

| 단계 | 주체 | 주요 활동 |
|------|------|----------|
| 1️⃣ 목적 정의 | 사람 | 보고서 유형, 독자, 목적, 핵심 메시지 결정 |

| 4️⃣ 검증 | 사람 + AI | 수치·법규·출처 3단계 팩트체크 |
| 5️⃣ 다듬기 | 사람 | 톤 조정, 경영진 관점 반영, 면책 문구 삽입 |

### 모듈 5 — 할루시네이션 방지와 팩트체크

> 35분 · 할루시네이션 유형, 팩트체크 3단계, 10대 체크리스트, RAG/Grounding, 보험업 보안

#### 학습 목표

1. 할루시네이션의 정의, 유형, 발생 원인을 이해하고 AI 답변의 경고 신호를 감지할 수 있다
2. AI가 생성한 수치·법규·통계·출처를 3단계로 팩트체크할 수 있다
3. 보고서 작성 시 적용할 수 있는 10개 체크리스트를 활용해 할루시네이션을 제거할 수 있다
4. RAG, Grounding, 출처 표시 등 기술적 대응 방법을 이해하고 선택적으로 적용할 수 있다

#### 할루시네이션의 4가지 유형

| 유형 | 설명 | KDB생명 예시 |
|------|-----|---------|
| **허위 사실 (Fabrication)** | 없는 정보를 만들어냄 | "보험업법 제250조의3"이 실제 존재하지 않음 |
| **왜곡 (Distortion)** | 실제 정보를 잘못 표현 | ○○보험 K-ICS 비율을 "250%"로 보고(실제 180%) |
| **오인용 (Misattribution)** | 출처·작성자를 틀리게 인용 | 실제로 없는 보험개발원 보고서 번호 생성 |
| **시점 오류 (Temporal Error)** | 시점을 잘못 기재 | "K-ICS 기본자본 규제 2025년 전면 시행"(실제 일정 상이) |

#### 할루시네이션 경고 신호

- AI가 지나치게 구체적인 수치(예: "2026년 3월 15일 보험업감독규정 개정에 의해...")를 자신 있게 제시
- 출처를 요구하면 애매한 답변 또는 없는 URL
- 같은 질문을 다시 물으면 답변이 미세하게 바뀜
- 상식적으로 가능성이 낮은 조합(예: 특정 보험사의 허위 손해율·K-ICS 수치)

#### 팩트체크 3단계 (Step 1)

```
[Step 1 — 출처 존재 확인]
1. AI 인용 법령 → law.go.kr, 금감원 보험업감독규정
2. AI 인용 논문·리포트 → 보험연구원(KIRI), Google Scholar, RISS
3. AI 인용 보험 통계 → 보험개발원(insure.or.kr), 생명보험협회(klia.or.kr)
4. AI 인용 기업 재무 → DART 공시 원문, 금감원 경영공시
5. AI 인용 K-ICS 비율 → 금감원 보험통계포털 원본 데이터
6. AI 인용 금리·환율 → 한은 ECOS
7. AI 인용 의료·건강 통계 → 국민건강보험공단, 보험개발원 경험생명표
8. AI 인용 뉴스 → 해당 언론사 사이트에서 직접 검색
```

#### 팩트체크 3단계 (Step 2) — 원문 대조

```
[대조 체크리스트]
□ 조항 번호(보험업법/감독규정) / 공시 번호(DART) / 리포트 ID(KIRI·보험개발원) 정확성
□ 조항·공시·리포트의 내용이 AI 요약과 일치
□ 수치(손해율, 사업비율, K-ICS 비율, 해약률, 수입보험료)가 원문과 같음
□ 적용 대상(상품군, 보험사, 기간) 올바름
□ 시행일·공시일·기준일 최신
```

#### 기술적 대응 — RAG / Grounding / 출처 표시

- **RAG**: 내부 보험계약DB·보험금지급DB·K-ICS 산출 데이터·내규를 AI에 연결 → 허위 생성 원천 차단
- **Grounding**: AI 답변을 특정 문서에 근거하도록 제한 (Claude의 Projects 기능, Gemini Grounding)
- **Perplexity 출처 인용**: 모든 주장에 인라인 링크 → 클릭으로 원문 즉시 확인

### 모듈 6 — KDB생명 실무 3대 시나리오

> 50분 · 종합 실전 적용 — 인수심사·보험시장분석·자산운용 3대 시나리오, Before/After, 보안 판단

#### 학습 목표

1. KDB생명 임직원의 실제 업무 상황에서 AI를 활용한 문제 해결 워크플로를 설계할 수 있다

아래 실제 데이터를 기반으로 인수심사 의견서 초안을 작성해주세요.

[사내 시스템에서 추출한 실제 데이터 — 붙여넣기]
- 피보험자 건강검진 결과 (혈압, 혈당, BMI, 간기능, 신장기능)

분석 순서는 아래 CoT(단계별 사고)를 따라주세요:

Step 1 — 의학적 리스크 분석

- 인수 조건(할증률/면책기간/부담보) 제안 포함
- 모든 수치는 입력된 실제 데이터 기반으로만 계산
- 위험률 데이터가 없는 경우 "보험개발원 위험률표 확인 필요"로 표시
```

**기법 해설:** Role-play + CoT + 구조화 출력 + Few-shot 최강 조합.

**검증 포인트:**
- 피보험자 의료기록 원문 대조 (건강검진 결과표)
- 보험개발원 위험률표·경험생명표 기준 초과위험도 확인
- 계약적부조사 기록 및 기존 보유계약 조회
- 직업위험등급 분류표 원본 확인

**보안 판단:**
- 피보험자 의료정보·건강정보·개인정보는 **사내 LLM 전용**
- 인수심사 중 건은 완료 시까지 외부 입력 절대 금지
- 일반적인 언더라이팅 원칙·위험률 개념은 외부 LLM 허용 (개인 식별정보 제거 후)

아래 실제 데이터를 기반으로 투자검토 보고서 초안을 작성해주세요.

[사내 시스템에서 추출한 실제 데이터 — 붙여넣기]
- 펀드 구조: 부동산 PF 펀드, 총 2,000억원

분석 순서는 아래 CoT(단계별 사고)를 따라주세요:

Step 1 — 투자구조 분석
  - 펀드 구조 (GP/LP, 선순위/후순위)
  - 기초자산 현황 (입지, 임대차 현황, 감정평가액)
  - 참여자 (운용사, 시공사, 임차인 신용도)

Step 2 — 수익률 분석
  - 배당수익률, IRR, 만기수익률
  - 보험부채 예정이율 대비 스프레드
  - 유사 펀드 성과 벤치마크

Step 3 — ALM 정합성
  - 보험부채 듀레이션 대비 자산 듀레이션 매칭
  - 금리 시나리오별 ALM 갭 분석
  - 유동성 매칭 (보험금 지급 패턴 vs 펀드 현금흐름)

Step 4 — K-ICS 영향 분석
  - 신용위험액 변동 (부동산 위험계수 적용)
  - 시장위험액 변동 (금리 리스크, 부동산 가격 리스크)
  - K-ICS 비율 변동 추정 (투자 전후 비교)

Step 5 — 리스크 매트릭스
  - 시장 리스크 (부동산 가격 하락, 금리 상승)
  - 신용 리스크 (임차인 부도, 공실 확대)
  - 유동성 리스크 (펀드 환매 제한, 매각 지연)
  - 운영 리스크 (건물 관리, 법적 이슈)

출력 형식:
| 분석 항목 | 기본 시나리오 | 스트레스 시나리오 | 최악 시나리오 |

출력 조건:
- 과거 검토한 유사 딜 양식(Few-shot으로 제공)을 따라주세요
- 투자 결정을 확정하지 말고 "검토 의견"으로만 제시
- K-ICS 비율 변동은 반드시 시나리오별 표로 제시
- 모든 수치는 입력된 실제 데이터 기반으로만 계산
- 시장 데이터가 없는 경우 "감정평가서/임대차 현황 확인 필요"로 표시
```

**기법 해설:** Role-play + CoT + 구조화 출력 + Few-shot 최강 조합. 특히 보험사 자산운용의 핵심인 ALM 정합성과 K-ICS 영향 분석을 별도 Step으로 분리하여 누락을 방지합니다.

**검증 포인트:**
- 감정평가서 원본 — 기초자산 가치 대조
- 임대차 현황표 — 임대율·주요 임차인 신용도 확인
- 유사 펀드 성과 — 운용사 제공 실적 vs 시장 벤치마크
- K-ICS 위험계수 — 금감원 고시 기준 부동산 위험계수 적용 확인
- ALM 데이터 — 보험부채 듀레이션 최신 산출 결과

**보안 판단:**
- 투자검토 중인 건의 상세 정보(펀드명·운용사·금액)는 **사내 LLM 전용**
- 투자심의위원회 상정 전까지 외부 LLM 사용 금지
- 일반적인 부동산 시장 동향·ALM 개념 질의는 외부 LLM 허용 (투자건 식별정보 제거 후)

| **금리 거짓값** | "국고채 10년물 3.5% (당일)" | Bloomberg / 한은 ECOS |

> **보험업 할루시네이션 3대 위험 지대**:
> 1. **보험업법 조항** — AI가 존재하지 않는 조·항을 자신있게 인용하는 경우가 빈번합니다
> 2. **경험생명표·질병률** — 보험개발원이 발표하지 않은 통계표를 만들어내는 경우가 있습니다
> 3. **K-ICS 비율·CSM 수치** — 특정 보험사의 재무건전성 수치를 부정확하게 제시하는 경우가 많습니다

### E. 실습 문제 세트

#### 모듈 1 실습

1. "보험 인수심사 해줘" 프롬프트의 3가지 함정을 식별하고 개선 버전 작성
2. 50대 남성 고액 건강보험 계약의 인수 리스크 분석용 CoT 프롬프트 작성 (5단계)
3. 본인 업무 1개 선정 → RTCF 프레임워크로 템플릿화

#### 모듈 2 실습

1. 동일 질문("건강보험 시장 2026 전망")을 ChatGPT / Claude / Perplexity에 각각 입력 → 결과 차이 비교표 작성
2. 보안 판단 플로차트로 본인 업무 데이터 3종 분류
3. 사내 LLM으로 보험 포트폴리오 분석 프롬프트 초안 작성

#### 모듈 3 실습

1. 본인 부서 표준 보고서 양식을 Few-shot 예시로 → AI에 적용 테스트
2. 일일 보험시장 브리핑 자동화 워크플로 설계
3. 자산운용 투자검토 템플릿을 실제 투자건에 적용해 초안 생성

#### 모듈 4 실습

1. 가상 상황("○○보험상품 보험업법 위반 가능성")을 Perplexity로 검색 → 출처 3개 원문 확인
2. 보험개발원 + 생명보험협회 데이터 교차로 경쟁사 건강보험 손해율 검증
3. 해외 보험시장 규제(ICS 2.0, EU Solvency II) 비교 분석 AI 초안 → 원문으로 3개 이상 검증

#### 모듈 5 실습

1. AI가 제시한 가상 보험업법 조항 3건 중 실존 1건, 허위 2건 구별
2. 보험개발원 원문 통계 1건 + AI 요약 1건 → 차이점 매트릭스 작성
3. 본인 최근 보고서 1건에 10대 체크리스트 전수 적용

#### 모듈 6 실습

1. 시나리오 1 프롬프트를 본인 업무 보험계약으로 치환 → 실제 생성 결과 비교
2. 시나리오 2를 연금보험 시장으로 치환 → 보험시장 분석 리포트 초안
3. 시나리오 3을 인프라 투자 건으로 치환 → 투자검토 보고서 초안

### F. 용어 사전

#### AI / 프롬프트 기법

| 용어 | 의미 |
|------|-----|
| **RTCF** | Role-Task-Context-Format. 구조적 프롬프트 4요소 |
| **Zero-shot** | 예시 없이 바로 요청하는 프롬프트 기법 |
| **Few-shot** | 예시 1~3개를 먼저 보여주는 프롬프트 기법 |
| **Role-play** | AI에게 전문가 역할을 부여하는 기법 |
| **CoT** | Chain of Thought. 단계별 추론을 요청하는 기법 |
| **RAG** | Retrieval-Augmented Generation. 외부 문서 검색 후 증강 생성 |
| **MCP** | Model Context Protocol. AI와 외부 시스템의 표준 연결 규격 |
| **Grounding** | AI 답변을 특정 자료에 근거하도록 제한하는 기술 |
| **Function Calling** | AI가 사전 정의된 함수를 호출하여 실행하는 기능 |
| **Agentic Workflow** | AI가 계획·실행·검증을 자율적으로 반복하는 워크플로 |
| **Human-in-the-loop** | 최종 판단·승인에 사람이 참여하는 구조 |
